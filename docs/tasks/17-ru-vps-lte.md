## 🎯 Цель

Завести в проде **VPS внутри РФ** (Yandex Cloud или Selectel) под inbound
`vless-reality-rusni-in` (1443/ads.x5.ru) — чтобы юзеры на мобильных LTE-
операторах с агрессивным DPI (Хакасия, Красноярский край, ХМАО, и т.д.)
получали туннель через российский IP-pool, который их операторы **не
блокируют** (т.к. это РУ→РУ внутри страны).

Текущий LTE-профиль через Hetzner DE Falkenstein **не работает в Хакасии**
несмотря на:
- Reality TLS handshake проходит (проверено в xray-логах сервера)
- TCP до :1443 не блочится (отдельные коннекты ходят)
- shortIds доработаны (`["b470aa0f3b156a0f", ""]` после 2026-05-07)

Гипотеза: **DPI оператора блочит iface-уровневый трафик к иностранным
IP-pool'ам** (Hetzner DE/FI как класс), не глядя на TLS-content. Это в
точности схема которая обходится «РУ-VPS в РФ→РУ маршруте».

## 📚 Контекст: разведка инфры конкурента (2026-05-07)

Юзер прислал 2 рабочих JSON-конфига конкурента (`geodataload.com`).
Реверс-инжиниринг показал:

### DNS-схема `geodataload.com`

- NS: `gerald.ns.cloudflare.com` / `penny.ns.cloudflare.com` (бесплатный CF DNS, **не proxy** — A-records прямые)
- TTL поддоменов: ~215 сек (быстрая ротация IP без обновления подписок)
- Round-robin: каждый поддомен возвращает **3-4 A-records**, клиент
  random'но идёт на любой из них; failover автоматический.

```
w1.geodataload.com → 51.250.4.191, 51.250.7.38, 51.250.10.154, 51.250.14.171  (4 IP)
11.geodataload.com → 144.31.131.232, .233, .242, .243                          (4 IP)
1.geodataload.com  → 144.31.3.4
2.geodataload.com  → 178.72.133.238
```

### Где реально стоят их серверы

| поддомены | назначение | IP-pool | AS / провайдер |
|---|---|---|---|
| **w1, w2, w3** | LTE-обход | `51.250.x.x`, `84.201.x.x`, `84.252.x.x` | **Yandex.Cloud RU** (AS200350 / RU-YANDEXCLOUD) |
| **2** | резерв | `178.72.x.x` | **Selectel RU** (AS49505 / SELECTEL-NET) |
| **1, 11, 12** | Wi-Fi/международный | `144.31.x.x`, `88.216.x.x` | RIPE NL 🇳🇱 / LT 🇱🇹 |

### Reality TLS-маскировка

Один whitelist-SNI для всей инфры:
```
w1.geodataload.com (Yandex RU :1443) → CN = ads.x5.ru
w2.geodataload.com (Yandex RU :1443) → CN = ads.x5.ru
1.geodataload.com  (RIPE NL :443)    → CN = ads.x5.ru
2.geodataload.com  (Selectel RU :443) → CN = ads.x5.ru
```

LE-сертификат R13 один, переиспользуется через Reality steal-from-real.
В клиентском конфиге Wi-Fi-1 они подсовывают `serverName = www.aviasales.com`
(несмотря на реальный CN=ads.x5.ru) — рискованная практика, не копируем.

### Reality `publicKey` reuse

`xsrPNGeq3ap9XVIztgtSJ7eg6Jl2ZWvSCRqZiXV2fTw` — **одинаковый** на всех VPS
(Yandex RU + Selectel + RIPE NL + LT). Они переиспользуют `privateKey`
между серверами — DevOps-удобство, но плохая security (компромисс одного
= компромисс всех). **Мы так делать не будем**, у нас уже per-server keys.

### Порты
- `1443` — LTE (нестандарт, не в популярных VPN-blocklist'ах)
- `443` — обычные/Wi-Fi
- `24231` — рандом (для Wi-Fi профилей)

Никакой системы кроме «что не блочится — то и используем».

### Шаг B — VPS в Yandex.Cloud RU 🟡 (этот тикет)

#### Выбор провайдера

| провайдер | плюсы | минусы | стоимость (1GB / 1vCPU / 20GB) |
|---|---|---|---|
| **Yandex Cloud** | Большой выбор регионов (Москва, СПб), стабильный API, IP-pool как у конкурента | RU-юр.лицо / ИП / самозанятый, привязка к РУ-карте | ~300 ₽/мес |
| **Selectel** | Проще регистрация (физ.лицо ОК), оплата картой Tinkoff | Меньше DC, шумные соседи на shared | ~250 ₽/мес |
| **VK Cloud** | Аналог Яндекса | Слабее SLA | ~280 ₽/мес |
| **Beget** | Самые дешёвые VPS, попсовый | Часто забанены за VPN | ~150 ₽/мес |

**Рекомендация: Yandex Cloud** (повторяем конкурента).

#### Регион

Москва (`ru-central1-a`) — большинство магистральных Tier-1 АЗ-маршрутов.
Альтернатива: `ru-central1-b` (Калуга) если в Москве шумно.

#### Спецификации

- **vCPU:** 1 (хватает для Reality + freedom outbound)
- **RAM:** 1 GB (Xray ~50 MB resident; запас на стабильность)
- **Disk:** 20 GB SSD (минимум; trafic не пишем на диск)
- **Network:** 100 Mbit (стандарт, бесплатный трафик включён)
- **OS:** Ubuntu 24.04 LTS (как у нас на остальных)

Ожидаемая стоимость: ~300 ₽/мес.

#### Настройка

1. **Создать VPS** в Yandex Cloud Console
2. **Скопировать `add_server/prepare-vps.sh`** на новую VPS, выполнить
   (создаёт docker, ip6tables, fail2ban, ssh hardening) —
   ⚠️ **сначала запатчить ip6tables-блок** (TODO из 2026-05-07)
3. **Сгенерить новые Reality keys** через `deploy/scripts/deploy-xray-new.sh`
   (НЕ переиспользовать ключи Falkenstein, в отличие от конкурента)
4. **Положить `xray.json`** с одним inbound:
   - `port: 1443`
   - `dest: ads.x5.ru:443`
   - `serverNames: ["ads.x5.ru"]`
   - `shortIds: ["<random_8byte_hex>", ""]` (сразу с empty для нового кода)
   - `tag: vless-reality-rusni-in` (тот же что на Falkenstein — сэкономит фикс bookkeeping-bug ниже)
5. **INSERT в БД** (через psql на backend):
   ```sql
   INSERT INTO vpn_servers (
     name, country_code, host, port, public_key, short_id, server_names,
     inbound_tag, xray_api_host, xray_api_port, priority, is_active
   ) VALUES (
     '[LTE 2] Россия (мобильный)',
     'RU',
     '<публичный IP Yandex VPS>',  -- или DNS-домен (см. Шаг C)
     1443,
     '<публичный_ключ>',
     '<short_id>',
     'ads.x5.ru',
     'vless-reality-rusni-in',
     '<публичный IP>',  -- или приватный если через VPN-туннель к backend
     10085,
     5,  -- priority выше id=144 (=10), чтобы LTE-2 шёл первым в подписке
     true
   );
   ```
6. **ResyncServer** через grpcurl:
   ```bash
   docker run --rm --network vpn-stack_vpn fullstorydev/grpcurl \
     -plaintext -d '{"server_id":<новый_id>}' \
     vpn-service:50062 vpn.v1.VPNService/ResyncServer
   ```
7. **Тест на Тапдыге:** обновить подписку в Happ, выбрать новый
   профиль `📶 [LTE 2] Россия (мобильный)`, попробовать тот же сценарий.
   Если работает — гипотеза «DPI режет иностранные IP-pool'ы»
   подтверждена.

#### Важно: юр.слой

- **Регистрация Yandex Cloud:** ИП или ООО (физ.лица не принимают, требуют ИНН).
  Если у aziz есть ИП — оформляется за 30 минут.
- **СОРМ / ФЗ-152 / закон 90-ФЗ:** Yandex Cloud по дефолту **не подключает
  СОРМ к VPS клиента** — это на ответственности клиента. Для VPN-сервиса
  с малым числом юзеров (<1000) РКН пока не требует, но риск есть.
  В случае запроса от ФСБ — Yandex выдаст логи (биллинг, IP-сессии),
  но не содержимое (не их юрисдикция, у них VPS-IaaS, не VPN-провайдер).
- **ToS Yandex Cloud:** «не использовать для нарушения российского
  законодательства». VPN сам по себе **не запрещён**; запрещено
  использовать VPN **для** обхода блокировок РКН-реестра. Тонкая
  граница — в случае жалобы могут заблокировать VPS без объяснений.
  Mitigation: в SLA-договоре прописать «защищённый удалённый доступ
  для сотрудников», как у обычных корп. VPN.
- **Backup-провайдер:** держать **аналогичный VPS на Selectel** как
  hot-spare на случай блокировки в Yandex.Cloud.

### Шаг C — DNS-домен в `address` (вместо IP) 🟡

После Шага B (когда есть RU-VPS), **завести DNS-доменное имя**:

- Поднять у Cloudflare (DNS-only, не proxy) запись `lte.osmonai.com` или
  `lte-ru.osmonai.com` → A-record на IP RU-VPS.
- В `vpn_servers.host` для нового сервера указать **домен**, не IP.

Преимущества:
1. **Ротация IP без update подписок:** если IP попал в blocklist оператора —
   меняем A-record, юзеры через ~5 мин (DNS TTL) уже на новом IP.
2. **Round-robin для HA:** когда станет 2+ RU-VPS, добавляем им оба IP
   как A-records одного домена → клиенты random'но распределяются,
   failover автоматически.
3. **Обходит часть DPI:** DPI операторов часто работает по «known VPN-
   IP-listам», IP можно ротейтить быстрее чем blocklist обновляется.

⚠️ **Минус:** DNS-resolution на стороне клиента. Happ резолвит через
**plain DNS 1.1.1.1/1.0.0.1** (мы это уже настроили в `buildPlainDNS`),
не через DoH (который оператор может зарезать). DNS-leak не критичен
здесь (домен публичный).

### Шаг D — упростить routing rules под конкурентский шаблон 🟡

Заменить наш `ruDirectDomains` (300 ручных доменов в `routing_lists.go`)
на короткий, но более полный список **только для priority-серверов**:

```go
ruDirectShort := []string{
    "domain:ru",
    "domain:рф",
    "domain:su",
    "domain:xn--p1ai",
    "geosite:private",
    "geosite:category-ru",
}
ruDirectShortIPs := []string{
    "geoip:ru",
    "geoip:private",
}
```

Это:
- **Покрывает РУ-зону шире** (любой `.ru` сайт, даже не из нашего списка)
- **Сходится с конкурентом** (одинаковая семантика split-tunnel)
- **Меньше maintenance** (геосайты обновляются автоматически с xray-core)

Применять только для priority-серверов в `profileBypass`. Обычный
`profileBypass` для не-priority остаётся со старым 300-доменным списком
чтобы не ломать поведение существующим юзерам без LTE.

Реализация: новая функция `buildRoutingPriorityRU(profile)` в
`subscription_config.go`, развилка в `writeJSONFormat` после `splitByPriority`.

### Шаг E — `[Wi-Fi 1]` нейтральный профиль (опционально) 🟡

Конкурент даёт `[Wi-Fi 1] 🇲🇩 Молдова` — для юзеров на «нормальном»
интернете без РУ-маскировки. У нас можно сделать аналог:

- Открыть на Hetzner DE-серверах **второй inbound** на нестандартном
  port (e.g. `:14443`) с SNI = `www.aviasales.com` (или другим
  нейтральным whitelist-доменом).
- Эмитить как `🇩🇪 [Wi-Fi 1] Германия` в подписке с `priority=15`
  (ниже LTE id=144 priority=10, выше каскада id=142 priority=20).
- Использовать `profileBypass` с обычным RU-routing (как сейчас у LTE).

⚠️ **Но**: у нас уже есть `⚡ Обычный VPN` который и есть «нейтральный
дефолт». Делать ли отдельный профиль `[Wi-Fi 1]` — вопрос UX, не
обхода. Возможно стоит отложить пока user feedback не появится.

## 🔥 Риски

| Риск | Вероятность | Удар | Mitigation |
|---|---|---|---|
| **Yandex.Cloud забанит VPS за VPN** | средняя (есть прецеденты) | потеря LTE-обхода, юзеры уходят | hot-spare на Selectel, готов скрипт миграции |
| **РКН включит Yandex.Cloud IP-pool в блок-лист** | низкая (это их же гос-облако) | потеря для не-LTE юзеров (но LTE-обход не нужен) | переезд на VPC-IP без публичного, через NAT |
| **Reality private key утечёт с одного VPS** | низкая | компрометация ОДНОГО сервера, не всех (per-server keys) | per-server keygen уже сделан |
| **СОРМ-запрос от ФСБ** | низкая (для масштаба ~200 юзеров) | Yandex выдаст биллинг + IP-сессии (не content) | договор оформить как «защищённый удалённый доступ» |
| **Юр.лицо заблокируют ФНС** | низкая | потеря Yandex-аккаунта | ИП оформлять не в РФ-резиденции (риск самозанятый из стран ЕАЭС) |
